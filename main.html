The Problem

Fraud Is Already Ahead

Traditional fraud detection is reactive — it reviews transactions after they complete. By then, money is gone and recovery is near impossible.

🤖

AI-Powered Social Engineering

Fraudsters use deepfake voice calls, AI-generated phishing emails, and real-time conversation coaching to manipulate victims into authorising transfers they believe are legitimate.

⏱️

Detection Happens Too Late

Current rule-based systems and batch ML models only flag transactions after settlement — sometimes hours later. The money has already left the account.

📊

High False Positive Rates

Existing solutions block too many legitimate transactions, damaging customer trust. Users get locked out of genuine payments, causing direct revenue loss.

🔍

No Contextual Intelligence

Static rules can't adapt to evolving fraud patterns. They miss novel attack vectors like geography jumps, device mismatches, and behavioural anomalies combined.

A request enters the system and traverses a multi-layer pipeline — completing the full fraud analysis and decision in under 75ms average.

01

JSON payload arrives via POST endpoint. Request validation, auth header check, rate limiting applied.

~10ms
02

Raw transaction enriched with computed features: device delta, geo velocity, amount z-score, time patterns.

~30ms

XGBoost Inference

Gradient-boosted ensemble model (pre-loaded in memory) produces fraud probability score from engineered features.

~10ms
04

Isolation Forest Anomaly Check

Unsupervised model detects statistically unusual transactions — catches novel fraud patterns outside training distribution.

~15ms

Threshold Decision Engine

If XGBoost ≥ 0.35 OR Isolation Forest anomalous → BLOCK. Otherwise → APPROVE. Response dispatched.

~10ms

ML Pipeline

Six-Step Engine

Built on the IEEE-CIS Fraud Detection dataset from Kaggle — real-world transaction data from Vesta Corporation with 590K records.

Step 01

Data Cleaning

Handle missing values, drop irrelevant columns, encode categoricals (card type, browser, device) into numeric representations the model understands.

# Encode card type + handle NaN
df.fillna(-999)
LabelEncoder().fit_transform(df['card4'])
Step 02

Feature Engineering

Create powerful derived features: time since last transaction, amount deviation from user average, geography velocity, and device mismatch flag.

# Geography velocity check
delta_time = tx['ts'] - prev_tx['ts']
geo_jump = haversine(loc1, loc2)
Step 03

Imbalanced Data Handling

Only ~3.5% of transactions are fraud. Naive training gets 96% accuracy while catching zero fraud. SMOTE and class weighting fix this.

# SMOTE oversampling
sm = SMOTE(random_state=42)
X_res, y_res = sm.fit_resample(X, y)
Step 04

XGBoost Training

Sequential gradient-boosted decision trees — each tree learns from the previous one's errors. Handles tabular data better than neural networks at this scale.

# XGBoost with class weight
model = XGBClassifier(
scale_pos_weight=97,
max_depth=6, n_estimators=500)
Step 05

Isolation Forest Layer

Unsupervised anomaly detection — no labels needed. Detects statistically unusual transactions regardless of whether fraud patterns appear in training data.

# Isolation Forest backup
iso = IsolationForest(contamination=0.035)
anomaly = iso.predict(X_test)
Step 06

Threshold Tuning

Default 50% confidence threshold creates too many false positives. Precision-recall curve analysis finds optimal threshold at ~35% for this use case.

# Threshold at 0.35 confidence
probs = model.predict_proba(X)[:,1]
preds = (probs >= 0.35).astype(int)
